# 🚀 azurelab-autopilot

> **PowerShell + Terraform autopilot** to deploy a Windows Server VM on Azure — fully interactive, no manual `.tfvars` editing required.

---

## 📋 Overview

`azurelab-autopilot` is a PowerShell-driven wizard that:

- Connects to Azure interactively
- Collects all VM parameters via **GUI dropdowns and prompts**
- Auto-generates `terraform.auto.tfvars`
- Runs `terraform init / validate / plan / apply` in one shot

No pre-filled config files. No hardcoded secrets. Just run and answer.

## 🗂️ Project Structure

```
azurelab-autopilot/
│
│   Run-AzureLabAutopilot.ps1       # Main entry point
│
└───terraform/
    │   main.tf                     # Root Terraform config
    │   variables.tf                # Input variable definitions
    │   outputs.tf                  # Outputs (IP, RG, etc.)
    │   .terraform.lock.hcl         # Provider version lock
    │
    └───modules/
        ├───networking/             # VNet, Subnet, NSG
        │       main.tf
        │       variables.tf
        │       outputs.tf
        │
        └───vm-test/                # Windows VM, NIC, Disks, Public IP
                main.tf
                variables.tf
                outputs.tf
```

> `.terraform/`, `terraform.auto.tfvars`, `*.tfstate` and `*.tfstate.backup` are excluded via `.gitignore`.

## ⚙️ Configuration

Open `Run-AzureLabAutopilot.ps1` and set your IDs at the top before first run:

```powershell
param(
  [string]$TenantId       = "your-tenant-id",
  [string]$SubscriptionId = "your-subscription-id",
  [string]$TerraformDir   = "C:\Projects\azurelab-autopilot\terraform"
)
```

Or pass them directly at runtime (see Usage below).

## 🚀 Usage

```powershell
# Option 1 — use defaults defined in script
```

## 🧭 Wizard Steps

Once launched, the script guides you through:

1. **Azure login** — `Connect-AzAccount` with your Tenant + Subscription
2. **Resource Group name** — Terraform will create it (warns if it already exists)
3. **VM name** and **Private IP** (static)
4. **Location** — searchable dropdown (e.g. `westeurope`)
5. **VM family** — A / B / D / DC family selector
6. **VM size** — filtered dropdown with vCPU and RAM info (e.g. `Standard_B2ms`)
7. **OS disk** — type (Standard_LRS / StandardSSD_LRS / Premium_LRS) + size in GB
8. **Data disk** — type + size (enter `0` to skip)
9. **Public IP** — yes/no prompt
10. **Admin credentials** — username + password (validated: 12+ chars, upper, lower, digit, special)
11. **Terraform action**:
    - `0` — generate tfvars only, no Terraform run
    - `1` — `init` + `plan`
    - `2` — `fmt` + `init` + `validate` + `plan` + `apply` (requires typing `APPLY` to confirm)

The script auto-generates `terraform/terraform.auto.tfvars` — **never commit this file**.

## 🖥️ What Gets Deployed

- **Resource Group**
- **Virtual Network** (10.0.0.0/16) with subnet `snet-test` (10.0.1.0/24)
- **Windows Server 2022 Datacenter Gen2** VM
- **Static private IP** NIC
- **OS disk** + optional **data disk**
- Optional **Public IP**
- **DNS servers**: VM private IP + Azure default (168.63.129.16)

## 🔒 Security Notes

- `terraform.auto.tfvars` contains your **admin password in plaintext** → excluded from git via `.gitignore`
- `*.tfstate` files may contain sensitive data → **never commit them**
- For production use, store state in a remote backend (Azure Storage Account)
- `*.json` files are broadly ignored — verify no legitimate config files are excluded in your case

## 📁 .gitignore Summary

| Pattern | Reason |
|---|---|
| `*.tfstate`, `*.tfstate.*` | Terraform state — may contain secrets |
| `*.tfvars`, `*.tfvars.json` | Generated by the script — contains credentials |
| `.terraform/` | Local provider cache — recreated by `terraform init` |
| `*.json` | Broad exclusion — review if you add JSON configs |
| `.env`, `secrets.json` | Credentials |
| `*.log`, `*.tmp` | Temp files |
